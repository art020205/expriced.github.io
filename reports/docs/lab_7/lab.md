# Отчет по лабораторной работе 3

## Тема

Упаковка FastAPI-приложения, базы данных и парсера данных в Docker. Вызов парсера из FastAPI напрямую по HTTP и через очередь задач Celery с Redis.

## Цель работы

Изучить контейнеризацию приложения с помощью Docker и Docker Compose, а также реализовать два способа взаимодействия FastAPI-приложения с сервисом парсинга:

- прямой HTTP-вызов отдельного parser-сервиса;
- асинхронный вызов через очередь задач Celery и брокер Redis.

В ходе работы необходимо разделить приложение на несколько сервисов, настроить их запуск в Docker Compose и обеспечить сохранение результатов парсинга в базу данных PostgreSQL.

## Структура проекта

Работа выполнена в папке `lr_3`.

```text
lr_3/
├── without_celery/
│   ├── app/
│   ├── parser/
│   └── docker-compose.yml
├── with_celery/
│   ├── app/
│   ├── parser/
│   └── docker-compose.yml
└── REPORT.md
```

Папка `without_celery` содержит реализацию прямого вызова парсера по HTTP.

Папка `with_celery` содержит реализацию вызова парсера через очередь задач Celery и Redis.

## Подзадача 1. Упаковка FastAPI, базы данных и парсера в Docker

### Описание

В первой части работы было создано отдельное FastAPI-приложение, которое содержит только эндпоинты, относящиеся к запуску парсеров.

Парсеры были вынесены в отдельные контейнеры. Для каждого подхода создан отдельный parser-сервис:

- `threading-parser` - парсер на основе `threading`;
- `multiprocessing-parser` - парсер на основе `multiprocessing`;
- `async-parser` - асинхронный парсер на основе `async` / `await` и `aiohttp`.

Все parser-сервисы используют один и тот же исходный код, но запускаются с разным значением переменной окружения `PARSER_MODE`.

### Сервисы Docker Compose

В `without_celery/docker-compose.yml` описаны следующие сервисы:

| Сервис | Назначение |
|---|---|
| `api` | Основное FastAPI-приложение, принимающее запросы от клиента |
| `threading-parser` | Контейнер парсера с режимом `threading` |
| `multiprocessing-parser` | Контейнер парсера с режимом `multiprocessing` |
| `async-parser` | Контейнер парсера с режимом `async` |
| `postgres` | База данных PostgreSQL |

### Dockerfile для основного приложения

Для FastAPI-приложения был создан Dockerfile в папке `without_celery/app`.

Он выполняет следующие действия:

- использует базовый образ `python:3.12-slim`;
- устанавливает зависимости из `requirements.txt`;
- копирует исходный код приложения;
- запускает приложение через `uvicorn`.

### Dockerfile для parser-сервиса

Для parser-сервиса был создан отдельный Dockerfile в папке `without_celery/parser`.

Он также использует образ `python:3.12-slim`, устанавливает зависимости и запускает FastAPI-приложение парсера на порту `8001`.

### База данных

Для хранения результатов используется PostgreSQL.

Parser-сервисы сохраняют данные в таблицу `parsed_pages`.

Основные поля таблицы:

| Поле | Описание |
|---|---|
| `id` | Уникальный идентификатор записи |
| `url` | URL страницы |
| `title` | Заголовок страницы или текст ошибки |
| `parser_type` | Тип парсера |
| `parsed_at` | Время сохранения результата |

Таблица создается автоматически при запуске parser-сервиса.

## Подзадача 2. Вызов парсера из FastAPI по HTTP

### Описание

В версии `without_celery` основное FastAPI-приложение принимает запрос от клиента и перенаправляет его в нужный parser-контейнер по HTTP.

Основное приложение не выполняет парсинг самостоятельно. Его задача - принять список URL, выбрать нужный parser-сервис и вернуть клиенту результат.

### Эндпоинты

| Метод | URL | Описание |
|---|---|---|
| `POST` | `/parsing/threading` | Запускает парсинг через контейнер `threading-parser` |
| `POST` | `/parsing/multiprocessing` | Запускает парсинг через контейнер `multiprocessing-parser` |
| `POST` | `/parsing/async` | Запускает парсинг через контейнер `async-parser` |

### Пример запроса

```json
{
  "urls": [
    "https://example.com",
    "https://www.python.org",
    "https://docs.python.org/3/",
    "https://fastapi.tiangolo.com/"
  ]
}
```

### Сценарий работы

1. Клиент отправляет POST-запрос в основное FastAPI-приложение.
2. FastAPI определяет, какой parser-сервис нужно вызвать.
3. FastAPI отправляет HTTP-запрос в соответствующий контейнер парсера.
4. Parser-сервис загружает HTML-страницы, извлекает заголовки и сохраняет данные в PostgreSQL.
5. Parser-сервис возвращает результат основному приложению.
6. Основное приложение возвращает результат клиенту.

### Запуск

```bash
cd lr_3/without_celery
docker compose up --build
```

После запуска Swagger-документация доступна по адресу:

```text
http://localhost:8000/docs
```

## Подзадача 3. Вызов парсера из FastAPI через очередь

### Описание

В версии `with_celery` был добавлен асинхронный запуск парсинга через очередь задач.

Для этого используются:

- `Celery` - система фоновой обработки задач;
- `Redis` - брокер сообщений и backend для хранения результатов задач;
- `celery-worker` - отдельный контейнер, который забирает задачи из очереди и выполняет их.

FastAPI больше не обязан ждать завершения парсинга. Он ставит задачу в очередь и сразу возвращает клиенту `task_id`.

### Сервисы Docker Compose

В `with_celery/docker-compose.yml` описаны следующие сервисы:

| Сервис | Назначение |
|---|---|
| `api` | Основное FastAPI-приложение |
| `celery-worker` | Celery worker, выполняющий задачи в фоне |
| `redis` | Брокер сообщений и хранилище результатов Celery |
| `threading-parser` | Parser-сервис с режимом `threading` |
| `multiprocessing-parser` | Parser-сервис с режимом `multiprocessing` |
| `async-parser` | Parser-сервис с режимом `async` |
| `postgres` | База данных PostgreSQL |

### Настройка Celery

В приложении был создан файл `app/celery_app.py`, в котором задается конфигурация Celery:

- Redis используется как broker;
- Redis используется как backend результатов;
- включено отслеживание состояния задач;
- задачи и результаты сериализуются в JSON.

Задача парсинга находится в файле `app/tasks.py`.

Она принимает:

- тип парсера;
- список URL.

После этого Celery worker вызывает нужный parser-контейнер по HTTP.

### Эндпоинты Celery

| Метод | URL | Описание |
|---|---|---|
| `POST` | `/parsing/celery/{parser_type}` | Ставит задачу парсинга в очередь Celery |
| `GET` | `/parsing/tasks/{task_id}` | Возвращает статус и результат задачи |

Допустимые значения `parser_type`:

- `threading`;
- `multiprocessing`;
- `async`.

### Пример постановки задачи

```bash
curl -X POST http://localhost:8000/parsing/celery/async \
  -H "Content-Type: application/json" \
  -d "{\"urls\":[\"https://example.com\",\"https://www.python.org\"]}"
```

Пример ответа:

```json
{
  "task_id": "a1b2c3d4-0000-1111-2222-abcdefabcdef",
  "status": "PENDING",
  "parser_type": "async",
  "message": "Parsing task has been sent to Celery queue"
}
```

### Пример получения результата

```bash
curl http://localhost:8000/parsing/tasks/a1b2c3d4-0000-1111-2222-abcdefabcdef
```

Возможный ответ во время выполнения:

```json
{
  "task_id": "a1b2c3d4-0000-1111-2222-abcdefabcdef",
  "status": "STARTED",
  "result": null,
  "error": null
}
```

Возможный ответ после завершения:

```json
{
  "task_id": "a1b2c3d4-0000-1111-2222-abcdefabcdef",
  "status": "SUCCESS",
  "result": {
    "parser_type": "async",
    "elapsed_seconds": 0.8497,
    "results": [
      {
        "id": 1,
        "url": "https://example.com",
        "title": "Example Domain",
        "parser_type": "async",
        "parsed_at": "2026-06-16T18:00:00"
      }
    ]
  },
  "error": null
}
```

### Сценарий работы через Celery

1. Клиент отправляет POST-запрос в FastAPI.
2. FastAPI создает Celery-задачу и отправляет ее в Redis.
3. FastAPI сразу возвращает клиенту `task_id`.
4. Celery worker забирает задачу из Redis.
5. Worker вызывает нужный parser-контейнер по HTTP.
6. Parser-контейнер выполняет парсинг и сохраняет результат в PostgreSQL.
7. Worker сохраняет результат выполнения задачи в Redis.
8. Клиент запрашивает результат по `task_id`.

### Запуск

```bash
cd lr_3/with_celery
docker compose up --build
```

После запуска доступны:

| Сервис | Адрес |
|---|---|
| FastAPI | `http://localhost:8000` |
| Swagger | `http://localhost:8000/docs` |
| PostgreSQL | `localhost:5433` |
| Redis | `localhost:6380` |

## Сравнение подходов

| Подход | Принцип работы | Преимущества | Недостатки |
|---|---|---|---|
| Прямой HTTP-вызов | FastAPI сразу вызывает parser-сервис и ждет ответ | Простая реализация, легко отлаживать | Клиент ждет завершения парсинга |
| Celery + Redis | FastAPI ставит задачу в очередь, worker выполняет ее в фоне | Не блокирует API, удобно для долгих задач, можно масштабировать worker'ы | Сложнее инфраструктура, нужен Redis и отдельный worker |

## Вывод

В ходе работы было реализовано контейнеризированное приложение, состоящее из FastAPI, PostgreSQL и отдельных parser-сервисов.

В версии `without_celery` парсеры вызываются напрямую по HTTP. Этот подход прост и подходит для быстрых операций.

В версии `with_celery` парсинг запускается через очередь задач Celery с использованием Redis. Такой подход лучше подходит для длительных операций, потому что основное API не блокируется на время выполнения задачи и может сразу вернуть клиенту идентификатор задачи.

Docker Compose позволил объединить все компоненты системы в единый оркестр сервисов и запускать приложение одной командой.
