# Отчет по лабораторной работе

## Тема

Различия между `threading`, `multiprocessing` и `async` в Python на примере вычислительной задачи и параллельного парсинга веб-страниц с сохранением данных в базу данных.

## Цель работы

Изучить три подхода к параллельному выполнению задач в Python:

- `threading` - выполнение задач в нескольких потоках;
- `multiprocessing` - выполнение задач в нескольких процессах;
- `async` / `await` - асинхронное выполнение задач в одном потоке с использованием цикла событий.

Также необходимо сравнить время выполнения программ и определить, для каких типов задач каждый подход подходит лучше.

## Задача 1. Подсчет суммы чисел

### Описание задачи

В первой части работы были реализованы три программы, которые вычисляют сумму всех чисел от `1` до `10000000000000`.

Каждая программа содержит функцию `calculate_sum()`, которая выполняет вычисления. Общий диапазон чисел разбивается на несколько частей, после чего каждая часть обрабатывается параллельно выбранным подходом.

Программы расположены в папке `lr_2`:

- `threading_sum.py` - реализация с использованием модуля `threading`;
- `multiprocessing_sum.py` - реализация с использованием модуля `multiprocessing`;
- `async_sum.py` - реализация с использованием `asyncio`;
- `run_all.py` - запуск всех вариантов и вывод результатов.

### Особенности реализаций

#### Threading

Вариант с `threading` использует несколько потоков. Каждый поток получает отдельный участок диапазона и вычисляет сумму для него. После завершения работы всех потоков частичные результаты объединяются.

Потоки хорошо подходят для задач ввода-вывода, но для CPU-bound задач в Python они ограничены GIL. В данной реализации вычисление суммы выполняется через формулу арифметической прогрессии, поэтому фактическая нагрузка на CPU небольшая.

#### Multiprocessing

Вариант с `multiprocessing` использует несколько независимых процессов. Каждый процесс получает свою часть диапазона и считает сумму отдельно. Такой подход позволяет обходить ограничение GIL, потому что каждый процесс имеет собственный интерпретатор Python.

Этот подход обычно лучше подходит для тяжелых вычислительных задач, но имеет накладные расходы на создание процессов и обмен данными между ними.

#### Async

Вариант с `asyncio` использует асинхронные задачи. Диапазон также делится на части, а каждая часть обрабатывается отдельной корутиной.

Асинхронный подход особенно эффективен для задач ввода-вывода, где программа часто ожидает ответ от сети, файла или базы данных. Для чистых вычислений `asyncio` не дает настоящего параллелизма по CPU, но в данном случае вычисления быстрые, так как используется формула суммы.

### Результаты выполнения

| Подход | Workers |   Elapsed, сек |   Total process time, сек |
|---|---:|---:|---:|
| `threading` | 8 | `0.001242` | `0.028074` |
| `multiprocessing` | 16 | `0.108769` | `0.152673` |
| `async` | 8 | `0.000113` | `0.081305` |

### Анализ результатов

Все три подхода вернули одинаковую сумму:

```text
50000000000005000000000000
```

Самое маленькое значение `Elapsed` показал асинхронный вариант. Это связано с тем, что вычисление суммы выполняется очень быстро и не требует настоящей длительной параллельной CPU-работы.

`threading` также показал малое время выполнения, так как вычисление по формуле почти мгновенное. При этом `Total process time` больше, чем `Elapsed`, потому что учитываются накладные расходы работы потоков и процесса.

`multiprocessing` оказался медленнее на этой задаче. Причина в том, что создание процессов и передача данных между ними занимают больше времени, чем само вычисление суммы по формуле. Для действительно тяжелых CPU-bound вычислений `multiprocessing` мог бы показать лучший результат, но для данной реализации накладные расходы оказались заметнее полезной работы.

## Задача 2. Параллельный парсинг веб-страниц

### Описание задачи

Во второй части работы были реализованы три программы для параллельного парсинга веб-страниц. Каждая программа содержит функцию `parse_and_save(url)`.

Функция выполняет следующие действия:

- загружает HTML-страницу по переданному URL;
- извлекает заголовок страницы из тега `<title>`;
- сохраняет результат в базу данных лабораторной работы 1;
- выводит результат на экран.

Для хранения результатов была добавлена отдельная таблица `parsed_pages`.

Основные поля таблицы:

| Поле | Описание |
|---|---|
| `id` | Уникальный идентификатор записи |
| `url` | URL страницы |
| `title` | Заголовок страницы |
| `parser_type` | Тип использованного парсера: `threading`, `multiprocessing` или `async` |
| `parsed_at` | Дата и время сохранения результата |

Файлы реализации расположены в папке `lr_1`:

- `threading_parser.py` - парсинг с использованием потоков;
- `multiprocessing_parser.py` - парсинг с использованием процессов;
- `async_parser.py` - асинхронный парсинг с использованием `aiohttp`;
- `parser_utils.py` - общие функции для разбиения URL, парсинга title и сохранения в БД.

В `main.py` добавлены отдельные эндпоинты:

| Эндпоинт | Назначение |
|---|---|
| `POST /parsing/threading` | Запуск парсинга через `threading` |
| `POST /parsing/multiprocessing` | Запуск парсинга через `multiprocessing` |
| `POST /parsing/async` | Запуск парсинга через `async` / `aiohttp` |

Пример тела запроса:

```json
{
  "urls": [
    "https://example.com",
    "https://www.python.org",
    "https://docs.python.org/3/",
    "https://fastapi.tiangolo.com/"
  ]
}
```

### Особенности реализаций

#### Threading

В реализации через `threading` список URL делится на части. Для каждой части создается отдельный поток. Потоки параллельно выполняют HTTP-запросы, парсят HTML и сохраняют результаты в базу данных.

Такой подход хорошо подходит для сетевых задач, так как во время ожидания ответа от сайта один поток может уступать выполнение другому.

#### Multiprocessing

В реализации через `multiprocessing` список URL также делится на части, но обработка выполняется в отдельных процессах. Каждый процесс независимо загружает страницы, парсит заголовки и сохраняет данные в БД.

Для парсинга веб-страниц этот подход обычно имеет лишние накладные расходы, потому что задача больше зависит от сети, чем от процессора.

#### Async

В реализации через `async` используется `aiohttp` и ключевые слова `async` / `await`. Запросы выполняются асинхронно в рамках цикла событий.

Такой подход хорошо подходит для большого количества сетевых запросов, потому что программа не блокируется на ожидании ответа от одного сайта и может сразу продолжать работу с другими запросами.

### Результаты выполнения парсинга

| Подход | Время выполнения, сек |
|---|---:|
| `threading` | `1.1201` |
| `multiprocessing` | `1.7609` |
| `async` | `0.8497` |

### Анализ результатов

Лучший результат при парсинге показал асинхронный подход:

```text
async: 0.8497 сек
```

Это ожидаемо, так как парсинг веб-страниц является I/O-bound задачей. Основное время программа тратит не на вычисления, а на ожидание сетевых ответов. `asyncio` и `aiohttp` позволяют эффективно обрабатывать несколько запросов без создания большого количества потоков или процессов.

`threading` также показал хороший результат. Потоки подходят для сетевых задач, потому что ожидание ответа от сайта освобождает выполнение для других потоков. Однако у потоков есть накладные расходы на создание и переключение контекста.

`multiprocessing` оказался самым медленным в задаче парсинга. Для сетевых запросов отдельные процессы обычно избыточны: создание процессов и обмен результатами добавляют накладные расходы, а реальной необходимости обходить GIL здесь нет.

## Итоговое сравнение подходов

| Подход | Лучше подходит для | Преимущества | Недостатки |
|---|---|---|---|
| `threading` | I/O-bound задачи | Простая реализация, хорошо подходит для сетевых запросов | Ограничен GIL для CPU-bound задач |
| `multiprocessing` | CPU-bound задачи | Реальный параллелизм на нескольких ядрах | Большие накладные расходы на процессы |
| `async` | Большое количество I/O-bound задач | Минимальные накладные расходы, высокая эффективность на сетевых операциях | Не ускоряет тяжелые CPU-вычисления само по себе |

## Вывод

В ходе работы были реализованы и сравнены три подхода к параллельному выполнению задач в Python.

Для задачи подсчета суммы все подходы вернули одинаковый результат. Из-за использования формулы арифметической прогрессии вычисления выполнялись очень быстро, поэтому основную роль играли накладные расходы выбранного подхода.

Для задачи парсинга веб-страниц лучший результат показал `async`, так как эта задача связана с ожиданием сетевых ответов. `threading` также оказался эффективным, а `multiprocessing` показал худшее время из-за лишних накладных расходов.

Таким образом:

- для сетевых задач удобнее использовать `async` или `threading`;
- для тяжелых вычислительных задач лучше подходит `multiprocessing`;
- выбор подхода зависит от типа задачи: CPU-bound или I/O-bound.
